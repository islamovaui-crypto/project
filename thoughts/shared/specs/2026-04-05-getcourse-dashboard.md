# GetCourse Analytics Dashboard — Спецификация

## Executive Summary

Внутренний дашборд для команды образовательного проекта (2–10 чел.), который подключается к GetCourse API, синхронизирует данные об участниках, заказах, прогрессе по урокам и анкетах, и отображает их в фильтруемых таблицах с выгрузкой в CSV. Включает AI-копилот для интерпретации данных.

---

## Problem Statement

Сейчас команда получает данные вручную через интерфейс GetCourse. Это медленно, не масштабируется при 6–20 продуктах и 1 000–10 000 участниках, не позволяет строить кросс-продуктовые срезы и исключать нерелевантных пользователей (сотрудники, тесты). Нет инструмента для быстрой интерпретации трендов.

---

## Success Criteria

- Команда получает актуальный срез данных за < 2 мин (нажал кнопку синхронизации → данные обновились)
- Можно отфильтровать участников конкретного продукта и исключить тестовые аккаунты за < 30 сек
- AI-копилот отвечает на вопрос по данным за < 10 сек
- Выгрузка CSV по текущему фильтру работает в один клик

---

## User Personas

**Куратор / методист** — не технический, смотрит прогресс конкретных студентов, хочет быстро найти "кто застрял на уроке 3".

**Руководитель проекта** — смотрит агрегаты: конверсия в оплату, процент дошедших до N урока, распределение ответов на анкеты.

**Аналитик** — строит срезы, выгружает CSV, задаёт вопросы копилоту.

---

## User Journey (основной сценарий)

1. Открывают дашборд → вводят общий пароль
2. Видят 4 вкладки: **Участники / Заказы / Прогресс / Анкеты**
3. Выбирают продукт из выпадающего списка (глобальный фильтр)
4. При необходимости нажимают "Синхронизировать с GetCourse" → индикатор загрузки → данные обновились
5. Применяют фильтры в нужной вкладке
6. При необходимости открывают панель AI-копилота и задают вопрос
7. Выгружают CSV

## Functional Requirements

### P0 — Must Have

#### Авторизация
- Защита единым паролем (bcrypt hash в env-переменной)
- Сессия через httpOnly cookie, срок 24 ч

#### Синхронизация с GetCourse
- Кнопка "Синхронизировать" запускает фоновый job
- Тянет через GetCourse Export API (асинхронный: запрос → polling → данные):
  - `/account/users` — профили участников
  - `/account/deals` — заказы и статусы оплат
  - `/account/payments` — платежи
  - `/account/groups` — список групп (для исключений)
- Данные сохраняются в локальную БД (PostgreSQL)
- Показывает время последней синхронизации

#### Прогресс по урокам и анкеты — два канала
**Канал 1 — Webhooks (для будущих событий):**
- Дашборд предоставляет endpoint `POST /api/webhook/getcourse`
- В GetCourse в разделе "Процессы" настраиваются операции "Вызов URL":
  - При прохождении урока → webhook с userId, lessonId, productId
  - При заполнении анкеты → webhook с userId, surveyId, ответами
- Данные записываются в БД в реальном времени

**Канал 2 — Загрузка CSV (для исторических данных):**
- Страница "Импорт" позволяет загрузить CSV-файлы, выгруженные из интерфейса GetCourse
- Поддерживаемые форматы: прогресс по урокам, ответы на анкеты
- Данные парсятся и сохраняются в БД (upsert)

#### Глобальный фильтр продукта
- Выпадающий список всех продуктов/курсов
- Применяется ко всем вкладкам одновременно

#### Исключение аккаунтов из статистики
- Раздел настроек "Исключённые аккаунты":
  - Вручную добавить ID / email
  - Указать теги GetCourse для автоисключения
  - Указать группы GetCourse для автоисключения
- Исключённые аккаунты не попадают ни в одну вкладку и не учитываются в агрегатах

#### Вкладка: Участники
- Таблица: ID, имя, email, телефон, теги, дата регистрации
- Фильтры: поиск по ID/email/имени, тег, группа
- Клик на строку → боковая панель с полным профилем + все его заказы + прогресс + ответы на анкеты

#### Вкладка: Заказы
- Таблица: ID заказа, участник, продукт, сумма, статус оплаты, дата
- Фильтры: статус (оплачен / не оплачен / отменён), продукт, дата
- Агрегаты вверху: всего заказов / оплачено / сумма оплат

#### Вкладка: Прогресс
- Таблица: участник, продукт, кол-во открытых уроков, кол-во пройденных, % выполнения, последняя активность
- Фильтры: по продукту, по диапазону % выполнения (например, 0–20%, застрявшие), по конкретному уроку (дошёл / не дошёл)
- Агрегаты: медианный % выполнения, % не начавших

#### Вкладка: Анкеты
- Список анкет (входные + внутрикурсовые) с выбором нужной
- Таблица: участник + его ответы по столбцам
- Фильтр по конкретному вопросу + конкретному ответу
- Возможность сравнить ответы разных сегментов

#### Выгрузка CSV
- Кнопка "Выгрузить CSV" в каждой вкладке
- Выгружает данные с учётом **текущих активных фильтров**
- Включает ID участника в каждой строке

### P1 — Should Have

#### AI-копилот (чат)
- Панель справа, открывается по кнопке
- Получает контекст: текущую вкладку + активные фильтры + агрегированные данные (не сырые строки — из соображений объёма токенов)
- Примеры запросов: "Почему упала конверсия?", "Кто из оплативших ни разу не открыл урок?", "Какой вопрос анкеты коррелирует с прохождением курса?"
- Powered by Claude API (claude-sonnet-4-6)

#### AI-автоинсайты
- После каждой синхронизации анализирует изменения
- Выводит карточки-инсайты на главной: "Конверсия в оплату упала на 15% за 7 дней", "45 участников застряли на уроке 3"
- Каждый инсайт кликабелен → открывает соответствующий фильтр

### P2 — Nice to Have

- Email-уведомления при критических аномалиях
- Сравнение когорт (заказавшие в разные периоды)
- История синхронизаций (что изменилось за последние N дней)

## Technical Architecture

### Stack
- **Frontend + Backend**: Next.js 14 (App Router) + TypeScript
- **БД**: PostgreSQL (через Prisma ORM)
- **Хостинг**: Vercel (фронт + API routes) + Supabase или Railway (PostgreSQL)
- **AI**: Claude API (`claude-sonnet-4-6`) через Anthropic SDK
- **Синхронизация**: Next.js API route запускает background job (через Vercel Background Functions или простой async процесс)

### Data Model (ключевые сущности)

```
User          — участник GetCourse (id, email, name, phone, tags, groups, excluded)
Order         — заказ (id, userId, productId, amount, status, createdAt)
Product       — курс/продукт (id, name, gcId)
LessonProgress — (userId, productId, lessonId, opened, completed, lastActivity)
Survey        — анкета (id, name, productId, type: onboarding|in-lesson)
SurveyAnswer  — (surveyId, userId, questionId, answer)
SyncLog       — (syncedAt, status, counts)
ExcludedAccount — (userId?, email?, tag?, group?)
```

### GetCourse API Integration
- Базовый URL: `https://{account}.getcourse.ru/pl/api/`
- Аутентификация: API-ключ в заголовке или query-параметре
- Export API работает асинхронно: запрос → получаем task_id → поллинг до готовности → скачиваем файл
- Лимиты: учитывать rate limiting, добавить retry с exponential backoff

### Security
- Пароль хранится как bcrypt hash в env
- API-ключ GetCourse только в серверных env-переменных, никогда на клиент
- Claude API ключ только на сервере
- Исключённые аккаунты фильтруются на уровне БД-запросов (не на клиенте)

---

## Non-Functional Requirements

- **Performance**: таблицы с пагинацией (50 строк), тяжёлые агрегаты считаются на сервере
- **Scalability**: до 10 000 участников × 20 продуктов — PostgreSQL справится без проблем
- **Reliability**: синхронизация идемпотентна (upsert, не insert), можно запускать многократно
- **Security**: данные участников (PII) только на собственном сервере, не утекают в внешние сервисы (кроме Claude — передаём только агрегаты)

---

## Out of Scope

- Мобильное приложение
- Автоматическая синхронизация по расписанию (только по кнопке)
- Редактирование данных участников (только чтение)
- Интеграция с другими платформами (только GetCourse)
- Публичный доступ / разные роли пользователей

---

## Open Questions for Implementation

1. **GetCourse API**: уточнить конкретные endpoint'ы для прогресса по урокам и анкетам на конкретном аккаунте — структура может отличаться от документации
2. **Объём данных за синхронизацию**: тянуть всё или только дельту (изменения за последние N дней)? Рекомендую дельту для скорости
3. **Claude контекст для копилота**: нужно определить, какие агрегаты передавать в системный промпт, чтобы не превышать лимиты токенов при 10 000 участниках
4. **Hosting**: выбрать между Vercel+Supabase (просто) и VPS (полный контроль)

## Appendix: Discovery Findings

- Текущий процесс: ручная выгрузка из интерфейса GetCourse
- Команда: 2–10 человек, один общий пароль достаточен
- Продуктов: 6–20, важна фильтрация по продукту как глобальный контекст
- Участников: 1 000–10 000
- Обновление данных: вручную по кнопке (не real-time)
- Критичная проблема: сотрудники и тестовые аккаунты искажают статистику — нужно исключение по тегам, группам и вручную
- AI-копилот: и чат по данным, и проактивные инсайты после синхронизации
